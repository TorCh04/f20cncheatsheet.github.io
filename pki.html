Network Security & Public Key Infrastructure (PKI) - Exam Revision

1. Security Services

  • Confidentiality
  • Integrity
  • Message Authentication
  • Nonrepudiation

Digital signatures provide integrity, authentication, and nonrepudiation.

2. Digital Signatures

  • Authenticate sender using private key.
  • Receiver verifies with sender's public key.
  • Verification function returns true or false.

3. Public Key Certificate

  • Issued and signed by a Certification Authority (CA).
  • Binds a subscriber's name to a public key.
  • Receiver verifies certificate using CA's public key.

4. Public Key Infrastructure (PKI)

  • Set of software, hardware, policies, and procedures for managing digital certificates.
  • Goal: Establish identity, control access, protect data, provide accountability.

Certificate Chains

  • Multiple CAs exist (national, commercial, corporate).

5. Trust Models

  • Third-party trust: Trust via a common CA.
  • Web of Trust: Direct trust, no central CA.
  • Hierarchical: Single root CA, all others below.
  • Distributed: Multiple CAs, basis for most internet certificates.

6. Certificate Lifecycle

  1. Creation: Certificate issued after identity verification.
  2. Suspension: Temporarily invalid.
  3. Revocation: Permanently invalid before expiry.
  4. Expiration: Certificate becomes invalid after expiry date.

7. Digital Certificate Standards

  • X.509: Used in browsers, S/MIME, IPsec, SSL/TLS.
  • PGP: Multiple CAs, web of trust, multiple user IDs.

8. OpenPGP

  • Standard for certificates, encrypted messages, private keys, signatures.

9. Web of Trust

  • PGP defines trust levels: Unknown, None, Marginal, Full.
  • Key validity: Valid, Marginally valid, Invalid.

10. Certificate Revocation

X.509

  • Certificate Revocation List (CRL)
  • Online Certificate Status Protocol (OCSP)

PGP

  • Signers revoke their own signatures.

11. Certification Authority (CA) Core Functions

  • Verify subject identity.
  • Sign certificates with private key.
  • Signature ensures certificate integrity.

12. CA Setup & Certificate Issuance

  1. Generate public/private key pair.
  2. Create X.509 certificate (self-signed for root CA).
  3. Generate Certificate Signing Request (CSR).
  4. CA verifies identity and issues certificate.
  5. Root CA's public key is pre-installed in OS/browsers.

13. PKI vs. Man-in-the-Middle (MITM) Attacks

  • Authentic certificate: Attacker can't decrypt secret.
  • Fake certificate: Not signed by trusted CA, browser warns user.
  • Attacker's certificate: Subject mismatch, browser terminates connection.

Always verify the common name matches the hostname!

14. Attack Surfaces on PKI

  • CA verification process (e.g., Comodo breach 2011)
  • CA signing process (e.g., DigiNotar breach 2011)
  • Algorithm weaknesses (e.g., MD5, SHA-1 collisions)
  • User confirmation flaws (e.g., Unicode phishing)

15. Key Exam Tips

  • Understand the difference between X.509 and PGP certificates.
  • Know the certificate lifecycle and revocation methods.
  • Be able to explain trust models and their real-world implications.
  • Recognize common PKI attack vectors and countermeasures.
  • Always check certificate subject (common name) matches the intended hostname.
